# GLM テストドキュメント

このドキュメントは、GLM (GitHub Life Manager) プロジェクトのテスト戦略とテストケースについて説明します。

## テスト方針

CLAUDE.md に記載されている通り、このプロジェクトでは以下のテスト方針に従っています：

- **手動テスト中心**: 主要な機能は手動でテストを実施
- **正常系フロー**: 各機能の期待される動作を確認
- **エラーハンドリング**: ネットワーク障害、認証エラー等の異常系を考慮
- **エッジケース**: 空のリスト、大量データ等の境界条件をテスト

## 自動テスト

### テストの実行方法

```bash
# 全テストを実行
cargo test

# 特定のモジュールのテストのみ実行
cargo test models
cargo test gh
cargo test config

# 詳細な出力で実行
cargo test -- --nocapture
```

## テストケース一覧

### models.rs のテスト

**Repository 構造体のテスト**

1. `test_repository_creation`
   - **目的**: Repository 構造体の基本的な作成をテスト
   - **種類**: 正常系
   - **内容**: 各フィールドが正しく設定されることを確認

2. `test_repository_with_none_description`
   - **目的**: 説明がない場合のリポジトリをテスト
   - **種類**: エッジケース
   - **内容**: description が None でも正常に動作することを確認

3. `test_repository_serialization`
   - **目的**: JSON シリアライズ/デシリアライズをテスト
   - **種類**: 正常系
   - **内容**: serde を使った変換が正しく行われることを確認

4. `test_repository_with_large_stars`
   - **目的**: 大量のスター数を持つリポジトリをテスト
   - **種類**: エッジケース
   - **内容**: 大きな数値でも正常に扱えることを確認

### gh.rs のテスト

**parse_repo_owner 関数のテスト**

1. `test_parse_repo_owner_valid_url`
   - **目的**: 正しい形式の URL からの抽出をテスト
   - **種類**: 正常系
   - **内容**: GitHub API の標準的な URL から owner と repo を抽出

2. `test_parse_repo_owner_trailing_slash`
   - **目的**: 末尾にスラッシュがある場合をテスト
   - **種類**: エッジケース
   - **内容**: 末尾のスラッシュの処理を確認

3. `test_parse_repo_owner_short_url`
   - **目的**: 短い形式の URL をテスト
   - **種類**: 正常系
   - **内容**: "owner/repo" 形式の URL からの抽出を確認

4. `test_parse_repo_owner_single_part`
   - **目的**: スラッシュがない単一パスの場合をテスト
   - **種類**: エッジケース
   - **内容**: None が返されることを確認

5. `test_parse_repo_owner_empty_string`
   - **目的**: 空文字列の場合をテスト
   - **種類**: エッジケース
   - **内容**: None が返されることを確認

6. `test_parse_repo_owner_with_dash_and_underscore`
   - **目的**: ダッシュやアンダースコアを含む名前をテスト
   - **種類**: 正常系
   - **内容**: 特殊文字を含むリポジトリ名の処理を確認

7. `test_parse_repo_owner_with_numbers`
   - **目的**: 数字を含むリポジトリ名をテスト
   - **種類**: 正常系
   - **内容**: 数値を含む名前の処理を確認

8. `test_parse_repo_owner_long_path`
   - **目的**: 長いパスでも最後の2つを抽出できることをテスト
   - **種類**: 正常系
   - **内容**: 長いパスでも正しく処理できることを確認

### config.rs のテスト

**AppConfig 構造体とファイル操作のテスト**

1. `test_app_config_default`
   - **目的**: デフォルト設定の作成をテスト
   - **種類**: 正常系
   - **内容**: AppConfig::default() が正しく動作することを確認

2. `test_app_config_with_repository`
   - **目的**: リポジトリ情報を持つ設定をテスト
   - **種類**: 正常系
   - **内容**: last_repository フィールドが正しく設定されることを確認

3. `test_app_config_serialization`
   - **目的**: 設定のシリアライズ/デシリアライズをテスト
   - **種類**: 正常系
   - **内容**: JSON 形式での保存と読み込みが正しく行われることを確認

4. `test_app_config_empty_serialization`
   - **目的**: 空の設定のシリアライズをテスト
   - **種類**: エッジケース
   - **内容**: 空の設定でも正常にシリアライズできることを確認

5. `test_load_config_with_nonexistent_file`
   - **目的**: 設定ファイルが存在しない場合の読み込みをテスト
   - **種類**: エッジケース
   - **内容**: 関数が正常に動作し、AppConfig を返すことを確認

## 手動テスト項目

以下の項目については、手動テストで確認します：

### 正常系フロー

1. **起動とリポジトリ選択**
   - アプリケーションが正常に起動する
   - リポジトリ一覧が表示される
   - キーボード操作でリポジトリを選択できる

2. **Issue 一覧表示**
   - 選択したリポジトリの Issue 一覧が表示される
   - 左パネルに Issue のリスト、右パネルに詳細が表示される
   - キーボード操作で Issue を選択できる

3. **Issue 作成**
   - `n` キーで新規 Issue 作成画面が表示される
   - 外部エディタが起動する
   - 作成した Issue が GitHub に反映される

4. **Issue 編集**
   - `e` キーで Issue 編集画面が表示される
   - 外部エディタが起動する
   - 編集内容が GitHub に反映される

### エラーハンドリング

1. **認証エラー**
   - GitHub CLI が未インストールの場合、分かりやすいエラーメッセージが表示される
   - トークンが無効な場合、適切なエラーメッセージが表示される

2. **ネットワーク障害**
   - API 接続失敗時に適切なエラーメッセージが表示される
   - レート制限に達した場合、ユーザーフレンドリーなメッセージが表示される

3. **入力エラー**
   - 不正な入力に対して適切に処理される
   - エディタのキャンセル時に適切に処理される

### エッジケース

1. **空のリスト**
   - リポジトリが0個の場合、適切なメッセージが表示される
   - Issue が0個の場合、適切なメッセージが表示される

2. **大量データ**
   - 多数のリポジトリがある場合でも正常に動作する
   - 多数の Issue がある場合でも正常に動作する
   - スクロールが正しく機能する

3. **特殊文字**
   - Issue のタイトルや本文に特殊文字が含まれる場合も正常に表示される
   - 日本語を含む Issue が正しく処理される

## テストの追加方針

新しい機能を追加する際は、以下のガイドラインに従ってテストを追加してください：

1. **純粋関数のテスト**: 副作用のない関数は積極的にユニットテストを追加
2. **エラーハンドリングのテスト**: エッジケースやエラー条件を網羅
3. **日本語コメント**: テストコードのコメントは日本語で記述
4. **手動テストドキュメント**: UI に関連する機能は手動テスト項目をこのドキュメントに追加
