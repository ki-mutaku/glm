# 機能仕様書: リポジトリスコープの Issue 管理

**機能ブランチ**: `001-repo-scoped-issues`  
**作成日**: 2025-01-22  
**ステータス**: ドラフト  
**入力**: ユーザー説明: "GitHubのプライベートリポジトリを選択して、そのリポジトリだけのIssueを閲覧・作成できるような機能"

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - リポジトリリストの閲覧 (優先度: P1)

開発者がアクセス可能なすべてのプライベートリポジトリを表示し、その Issue に集中するために 1 つを選択したい。

**この優先度の理由**: これは基盤となる機能です。リポジトリ選択がなければ、この機能の残りの部分は動作しません。ユーザーがワークフローを特定のリポジトリにスコープできるようにすることで、即座に価値を提供します。

**独立したテスト**: アプリを起動し、'r' キーを押して、リポジトリ名、説明、スター数を表示するナビゲート可能な項目でリポジトリリストが表示されることを確認することで、完全にテストできます。

**受け入れシナリオ**:

1. **前提条件** ユーザーがメイン Issue 画面を表示している、**操作** 'r' キーを押す、**結果** アクセス可能なプライベートリポジトリのリストを表示するリポジトリ選択画面が表示される
2. **前提条件** リポジトリリストが表示されている、**操作** ユーザーが 'j' または下矢印キーを押す、**結果** 選択がリスト内の次のリポジトリに移動する
3. **前提条件** リポジトリリストが表示されている、**操作** ユーザーが 'k' または上矢印キーを押す、**結果** 選択がリスト内の前のリポジトリに移動する
4. **前提条件** リポジトリがリスト内でハイライトされている、**操作** ユーザーが右パネルを表示する、**結果** 選択されたリポジトリの詳細情報（名前、説明、スター数）が表示される
5. **前提条件** リポジトリ選択画面が開いている、**操作** ユーザーが 'Esc' キーを押す、**結果** リポジトリコンテキストを変更せずにメイン Issue 画面に戻る

---

### ユーザーストーリー 2 - リポジトリスコープの Issue 表示 (優先度: P1)

開発者がリポジトリを選択し、そのリポジトリに属する Issue のみを表示したい。

**この優先度の理由**: これが核となる価値提案です。リポジトリコンテキストによる Issue のフィルタリング。P1 実装後すぐにテスト可能で、独立して価値があります。

**独立したテスト**: リストからリポジトリを選択し、Enter を押して、そのリポジトリの Issue のみが Issue リストに表示されることを確認することで、完全にテストできます。

**受け入れシナリオ**:

1. **前提条件** リポジトリリスト内でリポジトリがハイライトされている、**操作** ユーザーが Enter キーを押す、**結果** アプリが選択されたリポジトリの Issue のみを表示する Issue ビューに切り替わる
2. **前提条件** リポジトリが選択されている、**操作** Issue ビューが表示される、**結果** UI ヘッダーに現在選択されているリポジトリ名が表示される
3. **前提条件** リポジトリが選択されていない、**操作** ユーザーが Issue 画面を表示する、**結果** アプリが「My Issues」を表示し、ユーザーに割り当てられたすべての Issue を表示する（既存の動作）
4. **前提条件** オープンな Issue がないリポジトリが選択されている、**操作** Issue ビューが読み込まれる、**結果** Issue が存在しないことを示す空の状態メッセージが表示される
5. **前提条件** リポジトリが選択され Issue が表示されている、**操作** ユーザーが 'r' を押して別のリポジトリを選択する、**結果** Issue リストが新しく選択されたリポジトリの Issue のみを表示するように更新される

---

### ユーザーストーリー 3 - 選択したリポジトリでの Issue 作成 (優先度: P2)

開発者が、現在選択されているリポジトリにシンプルなフォームを使用して新しい Issue を作成したい。

**この優先度の理由**: Issue 作成は読み取り専用の閲覧体験に書き込み機能を追加します。閲覧よりも重要度は低いですが、実装されると完全なワークフローを提供します。

**独立したテスト**: リポジトリを選択し、'n' を押し、Issue フォーム（タイトルと本文）に記入し、Issue が GitHub に表示されることを確認することで、完全にテストできます。

**受け入れシナリオ**:

1. **前提条件** リポジトリが選択されている、**操作** ユーザーが 'n' キーを押す、**結果** タイトルと本文のフィールドを持つ新しい Issue 作成フォームが表示される
2. **前提条件** Issue 作成フォームが開いている、**操作** ユーザーがタイトルと本文を入力して送信する、**結果** GitHub API を介して選択されたリポジトリに Issue が作成される
3. **前提条件** Issue 作成フォームが開いている、**操作** ユーザーが本文フィールドを編集する必要がある、**結果** 既存の外部エディタ統合（既存コードから再利用）がリッチテキスト編集のために開く
4. **前提条件** Issue 作成フォームが開いている、**操作** ユーザーが 'Esc' キーを押す、**結果** Issue を作成せずにフォームが閉じ、Issue ビューに戻る
5. **前提条件** リポジトリが選択されていない、**操作** ユーザーが 'n' キーを押す、**結果** アプリが最初にリポジトリを選択する必要があることを示すエラーメッセージを表示する
6. **前提条件** 新しい Issue が正常に作成された、**操作** フォームが閉じる、**結果** Issue リストが更新されて新しく作成された Issue が表示される

---

### エッジケース

- アクセス可能なプライベートリポジトリがない場合はどうなりますか？
  - GitHub アクセスを確認するための指示を含む空の状態メッセージを表示
- GitHub API が利用できない、またはエラーを返す場合はどうなりますか？
  - UI にエラーメッセージを表示し、ユーザーが再試行できるようにする
- リポジトリに 100 を超える Issue がある場合はどうなりますか？
  - ページネーションを実装するか、「さらに読み込む」オプションで初期読み込みを制限する
- Issue 作成が失敗した場合（ネットワークエラー、API エラー）はどうなりますか？
  - API からのエラーメッセージを表示し、入力を失わずにユーザーが再試行できるようにする
- ユーザーの GitHub トークンが無効または期限切れの場合はどうなりますか？
  - 認証エラーを表示し、`gh auth login` を再実行するようユーザーに指示する
- 選択と Issue 作成の間にリポジトリがアーカイブまたは削除された場合はどうなりますか？
  - リポジトリが利用できなくなったことを示すエラーメッセージを表示する

## 要件 *(必須)*

### 機能要件

- **FR-001**: システムは GitHub API を介して認証されたユーザーがアクセス可能なすべてのプライベートリポジトリのリストを取得して表示しなければならない
- **FR-002**: システムはリポジトリリストビューで名前、説明、スター数を含むリポジトリの詳細を表示しなければならない
- **FR-003**: システムはキーボードコントロール（上下ナビゲーションに j/k、選択に Enter、キャンセルに Esc）を使用してリポジトリリストをナビゲートできるようにしなければならない
- **FR-004**: システムはリポジトリが選択されているときに、現在選択されているリポジトリからの Issue のみを表示するように Issue リストをフィルタリングしなければならない
- **FR-005**: システムはリポジトリコンテキストがアクティブなときに、UI ヘッダーに選択されたリポジトリ名を表示しなければならない
- **FR-006**: システムはリポジトリが選択されていないときに、既存の「My Issues」ビュー（ユーザーに割り当てられたすべての Issue）を維持しなければならない
- **FR-007**: システムはメイン Issue ビューからリポジトリ選択画面を開くためのキーボードショートカット（'r'）を提供しなければならない
- **FR-008**: システムは Issue 作成フォームを開くためのキーボードショートカット（'n'）を提供しなければならない
- **FR-009**: システムは GitHub API を介してユーザーが提供したタイトルと本文で選択されたリポジトリに新しい Issue を作成しなければならない
- **FR-010**: システムは作成中に Issue 本文コンテンツを編集するために既存の外部エディタ統合を再利用しなければならない
- **FR-011**: システムはリポジトリが選択されていないときに Issue 作成を防止し、適切なエラーメッセージを表示しなければならない
- **FR-012**: システムは新しい Issue を正常に作成した後に、新しいアイテムを反映するために Issue リストを更新しなければならない
- **FR-013**: システムはすべての GitHub API 呼び出しに既存の認証メカニズム（gh auth token）を使用しなければならない
- **FR-014**: システムは API エラーを適切に処理し、ユーザーフレンドリーなエラーメッセージを表示しなければならない
- **FR-015**: システムはすべての新しい画面でキーボード駆動のナビゲーションを維持しなければならない（マウスは不要）

### 主要エンティティ

- **リポジトリ（Repository）**: 名前（owner/repo 形式）、説明、スター数、プライベート/パブリックステータスを含む属性を持つ GitHub リポジトリを表します。関係: 複数の Issue を含みます。
- **Issue**: タイトル、本文、番号、状態（オープン/クローズ）、担当者を含む属性を持つ GitHub Issue を表します。関係: 1 つのリポジトリに属します。
- **リポジトリ選択状態（Repository Selection State）**: 現在のユーザーコンテキストを表し、現在どのリポジトリ（もしあれば）が選択されているかを追跡します。Issue ビューをフィルタリングし、新しい Issue 作成のターゲットを決定するために使用されます。

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: ユーザーはメイン画面からリポジトリ選択画面に 2 秒未満で移動できる（1 回のキー押下 + API レスポンス時間）
- **SC-002**: ユーザーはキーボードナビゲーションを使用して、50 以上のリポジトリのリストから 10 秒未満でリポジトリを閲覧して選択できる
- **SC-003**: Issue リストフィルタリングはリポジトリ選択に 2 秒未満で応答する（API 取得 + UI 更新）
- **SC-004**: ユーザーは 'n' を押してからリストに Issue が表示されるまで 30 秒未満でタイトルと本文を含む新しい Issue を作成できる
- **SC-005**: 100% のキーボードショートカット（'r'、'n'、'Esc'、'j'、'k'、'Enter'）がすべての新しい画面で一貫して動作する
- **SC-006**: あらゆるワークフロー（リポジトリ選択、Issue 閲覧、Issue 作成）を完了するためにマウス操作はゼロで済む
- **SC-007**: エラーメッセージは API 障害から 1 秒以内に表示され、問題と推奨されるアクションを明確に示す

## 前提条件

- ユーザーは既に `gh auth login` を使用して GitHub で認証しており、有効なトークンを持っている
- ユーザーは GitHub 上で少なくとも 1 つのプライベートリポジトリにアクセスできる
- 既存の octocrab 依存関係（バージョン 0.38）は、必要なすべての GitHub API 操作（リポジトリのリスト表示、Issue のリスト表示、Issue の作成）をサポートしている
- 既存の外部エディタ統合は、Issue 本文編集のために変更なしで再利用できる
- GitHub API 呼び出しにはネットワーク接続が利用可能（オフラインモードはスコープ外）
- 既存の TUI フレームワーク（ratatui）は、リポジトリリストと Issue 作成フォームを表示するための十分なウィジェットを提供している
- リポジトリリストのページネーションは、アクセス可能なすべてのリポジトリを事前に読み込むことで処理される（遅延読み込み/無限スクロールは v1 のスコープ外）
- 選択リストにはプライベートリポジトリのみを表示する必要がある（パブリックリポジトリはデフォルトで除外）
- Issue 作成にはタイトルと本文フィールドのみが必要。高度な機能（ラベル、マイルストーン、担当者）は v1 のスコープ外
- 一貫性のために既存のキーボードナビゲーションパターン（上下に j/k）が維持される
